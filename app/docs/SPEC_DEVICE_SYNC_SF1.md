# SF1: Rust Image Byte Commands

Three commands enabling image file transfer over the message channel: read a stored image as base64, write one from base64, and check existence. Rust-only; follows the one-file-per-command convention and the existing `commands/images/` patterns (`save_image.rs` is the reference for path construction and validation).

## Command Contracts

All three: validate `extension` against the same `["jpg", "jpeg", "png", "webp", "gif"]` list as `save_image.rs`, resolve `<app_data_dir>/images/{id}.{extension}` with the existing path pattern, return `Result<_, String>` with descriptive `map_err` messages.

- `save_image_bytes(app_handle, id: String, extension: String, data_base64: String) -> Result<(), String>` — decode base64, `fs::create_dir_all` the images dir, write the file (full overwrite).
- `read_image_bytes(app_handle, id: String, extension: String) -> Result<String, String>` — read the file, return its base64 encoding. `Err` when the file does not exist (the TS caller treats that as "cannot serve this file-request" and drops the request silently).
- `image_file_exists(app_handle, id: String, extension: String) -> Result<bool, String>` — `Ok(path.exists())`.

Verify the current `base64` crate's encode/decode API against its docs.rs page for the version cargo resolves before writing the calls, and record the verified API in `.claude/knowledge/` (no entry exists yet).

No same-SF consumers. SF4's `syncService` invokes all three (`save_image_bytes` on completed chunk assembly, `read_image_bytes` on incoming `file-request`, `image_file_exists` after images-row applies).
