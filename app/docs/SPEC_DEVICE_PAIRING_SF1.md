## Identity (`identity.rs`)

Load-or-create the device secret key from `<app_data_dir>/connectivity/device.key`. File content: the 64-char lowercase hex encoding of `SecretKey::to_bytes()`. On first run: `SecretKey::generate()`, create the directory, write the file. On subsequent runs: read, decode hex, `SecretKey::from_bytes(&[u8; 32])`. Use the existing app-data-dir pattern from `src-tauri/CLAUDE.md` (Common Patterns). All errors map to descriptive `String`s per the error convention.

## Endpoint Construction (`connections.rs`)

Build exactly this configuration (method names verified [S_1, S_2 in .claude/knowledge/iroh.md]):

```rust
Endpoint::builder()
    .secret_key(secret_key)
    .relay_mode(RelayMode::Disabled)
    .clear_address_lookup()
    .address_lookup(MdnsAddressLookup::builder())
    .alpns(vec![ALPN_MAIN.to_vec(), ALPN_PAIRING.to_vec()])
    .bind()
```

with `const ALPN_MAIN: &[u8] = b"gm-tool";` and `const ALPN_PAIRING: &[u8] = b"gm-tool-pairing";` in `connectivity/mod.rs`. If the installed iroh 1.0.2 builder requires a preset argument to `Endpoint::builder()`, resolve against docs.rs and record the finding — the invariant that must hold regardless of exact builder syntax: relays disabled, no DNS/pkarr publishing, mDNS as the only address lookup.

## State (`mod.rs`)

`ConnectivityState` is the managed Tauri state (registered in `lib.rs`), following the `PendingInstallState` pattern (`src-tauri/CLAUDE.md` — State Management) extended for async: wrap the inner data so commands and spawned tasks can share it (`Arc` + `tokio::sync::Mutex` via `tauri::async_runtime`). Inner data:

- initialized endpoint handle + own EndpointId (`None` until `init_connectivity`)
- own device name: `Option<String>` (cache for pairing hellos; source of truth is the TS `_system` row)
- trusted peers: set of EndpointIds
- live connections: map EndpointId → message sender handle
- pairing state: `Option<PairingSession>` — active code (`String`, 6 digits), open candidate connections, per-candidate failed-attempt counters

Only the state type alias and what commands need is re-exported from the module barrel; internals stay private, mirroring `PendingInstallState`/`PendingInstall`.

## Wire Protocol

Both ALPNs carry newline-delimited JSON frames over a single bidirectional stream per connection. Two distinct frame vocabularies:

**Pairing frames (`gm-tool-pairing`, defined in `pairing.rs`, serde with `#[serde(rename_all = "camelCase")]`):**

- `PairingHello { endpoint_id: String, name: Option<String> }` — sent by both sides immediately on pairing-connection establishment
- `CodeSubmit { code: String }` — sent by the side whose user typed the code
- `CodeVerdict { accepted: bool }` — reply from the verifying side

**Application frames (`gm-tool`):** each frame is one envelope JSON string, relayed verbatim between the `send_message` command and the `connectivity-message-received` event. Rust never parses envelope contents (KAD: Rust is a secure pipe).

## Behavior Contracts

**Accept loop:** for every incoming connection, branch on ALPN. `gm-tool`: if the remote EndpointId is not in the trusted set, close immediately; otherwise register the connection, emit `connectivity-peer-connected`, and pump incoming frames to `connectivity-message-received` events until the connection closes (then emit `connectivity-peer-disconnected` and deregister). `gm-tool-pairing`: if no pairing session is active, close immediately; otherwise run the pairing exchange below.

**Discovery/auto-connect:** subscribe to mDNS discovery events. On a discovered EndpointId in the trusted set with no live connection: dial with `ALPN_MAIN` only if own id is lexicographically smaller than the peer's (KAD: connection deduplication); the larger side waits to be dialed. On a discovered id not in the trusted set while a pairing session is active: dial with `ALPN_PAIRING` and run the pairing exchange as the probing side.

**Pairing exchange:** both sides send `PairingHello` on establishment. On receiving the peer's hello, emit `connectivity-pairing-candidate { endpointId, name }` and hold the connection open. When a `CodeSubmit` frame arrives: compare against the active session's own displayed code; reply `CodeVerdict`; on `accepted: true` both sides add the peer to the trusted set, emit `connectivity-pairing-succeeded { endpointId, name }`, and close the pairing connection (the normal auto-connect rule then establishes the `gm-tool` connection). On a wrong code, increment the candidate's failure counter; after 3 failures close the candidate connection and emit `connectivity-pairing-failed { endpointId, reason: "attempt limit reached" }`. If a candidate's connection drops before success, emit `connectivity-pairing-candidate-lost { endpointId }`.

**Exiting pairing mode** (command or app shutdown): clear the session and close all candidate connections.

## Commands

All follow `Result<T, String>` and the one-file-per-command convention. Signatures (Rust side; TS calls use camelCase keys per the root spec KAD):

- `init_connectivity(app_handle: AppHandle, state: State<'_, ConnectivityState>, own_name: Option<String>, trusted_peers: Vec<TrustedPeer>) -> Result<String, String>` — `TrustedPeer { endpoint_id: String, name: Option<String> }` (serde camelCase). Idempotent: when already initialized, update own name + trusted set and return the existing id without re-binding (KAD: StrictMode). Otherwise: load identity, build endpoint, store state, spawn accept loop + discovery subscription, return own EndpointId as 64-char hex.
- `enter_pairing_mode(state) -> Result<String, String>` — error if not initialized. Generate a 6-digit code via `rand` (range 0..=999999, formatted zero-padded to 6 chars), create the pairing session, return the code. Idempotent per session: if a session is already active, return its existing code.
- `exit_pairing_mode(state) -> Result<(), String>` — clears the session; no-op when none active.
- `submit_pairing_code(state, endpoint_id: String, code: String) -> Result<(), String>` — send `CodeSubmit` over that candidate's pairing connection and await the `CodeVerdict` with a 10-second timeout. `Ok(())` on accepted; `Err` with a descriptive message on rejection, timeout, or unknown candidate.
- `send_message(state, endpoint_id: String, envelope: String) -> Result<(), String>` — write one frame to the peer's live `gm-tool` connection; `Err` when not connected.
- `remove_trusted_peer(state, endpoint_id: String) -> Result<(), String>` — remove from trusted set, close any live connection to that peer. Does not send anything (the `unpair` envelope is TS's job, sent before this call).
- `get_connected_peers(state) -> Result<Vec<String>, String>` — EndpointIds with live `gm-tool` connections.
- `update_own_name(state, name: Option<String>) -> Result<(), String>` — refreshes the cached own name used in pairing hellos after a rename; error when not initialized. The persisted name lives in TS (`_system`); this cache only feeds `PairingHello` frames.

## Events

Emitted via the `Emitter` trait on `AppHandle` [S_4 in .claude/knowledge/tauri.md]. Payload structs in `connectivity/mod.rs`, serde `rename_all = "camelCase"`. Event name constants live beside them; TS mirrors these names in SF3 — the strings must match exactly:

| Event | Payload |
| --- | --- |
| `connectivity-peer-connected` | `{ endpointId: string }` |
| `connectivity-peer-disconnected` | `{ endpointId: string }` |
| `connectivity-message-received` | `{ endpointId: string, envelope: string }` |
| `connectivity-pairing-candidate` | `{ endpointId: string, name: string \| null }` |
| `connectivity-pairing-candidate-lost` | `{ endpointId: string }` |
| `connectivity-pairing-succeeded` | `{ endpointId: string, name: string \| null }` |
| `connectivity-pairing-failed` | `{ endpointId: string, reason: string }` |

Consumers: SF3 declares the TS-side contracts; SF5's lifecycle hook and SF7's dialog subscribe.
