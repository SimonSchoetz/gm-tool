# Release Pipeline Architecture

This document covers the CI/CD architecture for multi-platform, multi-channel releases. For the manual steps to cut a release (version bump, changelog, tagging), see `release-process.md`.

## Channel model

Stable and beta are **two separate Tauri apps with different identifiers**. They are not the same binary with a flag — they are independent installations.

This is required because the Tauri data directory is scoped to the app identifier. If both channels shared an identifier, they would share `%APPDATA%\{identifier}\` on Windows, including the SQLite database. A beta schema migration would then run against the stable database. There is no rollback mechanism, so this is unrecoverable.

With different identifiers:

- Separate install paths
- Separate data directories
- Separate SQLite databases
- Separate auto-updater endpoint (compiled into each binary at build time)
- Both can be installed on the same machine simultaneously

Data entered in beta is not visible in stable and vice versa. This is a known, accepted constraint.

| | Stable | Beta |
|---|---|---|
| Tauri identifier | `com.gm-tool` (current value) | `com.gm-tool.beta` |
| App name | GM-Tool | GM-Tool Beta |
| GitHub release type | Release | Prerelease |
| Auto-updater endpoint | stable releases endpoint | prerelease releases endpoint |

## Tag naming convention

All tags are pushed on `main`. There is no `stable-release` or `beta` long-running branch.

- `v1.0.0` → triggers stable release build
- `v1.0.0-beta.1` → triggers beta release build

Stability is determined entirely by the tag name, not the branch. The "Verify tag is on main" CI step in `release.yml` enforces that stable tags are only ever cut from `main`.

## Required configuration

### tauri.conf.json

`bundle.createUpdaterArtifacts` must be `true` for Tauri to generate `.sig` signature files alongside the installer bundles. Without it, signing is silently skipped and `latest.json` is never uploaded regardless of whether `TAURI_SIGNING_PRIVATE_KEY` is set.

```json
{
  "bundle": {
    "createUpdaterArtifacts": true
  }
}
```

`productName` must not contain spaces. tauri-action normalises filenames using dots but looks up signature files using the original name, causing a mismatch that silently prevents `latest.json` from being generated. Use a hyphen instead: `"GM-Tool"` not `"GM Tool"`.

### package.json

A `"tauri"` script must be present. tauri-action always invokes `npm run tauri build` — without this script the build fails immediately:

```json
{
  "scripts": {
    "tauri": "tauri"
  }
}
```

### GitHub Secrets

Two secrets must be set in the repository (Settings → Secrets and variables → Actions):

- `TAURI_SIGNING_PRIVATE_KEY` — private key generated by `npx tauri signer generate`
- `TAURI_SIGNING_PRIVATE_KEY_PASSWORD` — password used during key generation (empty string if no password was set)

The public key produced during generation must match the `plugins.updater.pubkey` value in `tauri.conf.json`. If the key is regenerated, both the secret and the pubkey must be updated together.

## Tauri build configuration

Two `tauri.conf.json` files:

- `app/src-tauri/tauri.conf.json` — stable builds, used by default
- `app/src-tauri/tauri.beta.conf.json` — beta overrides: `identifier`, `productName`, updater endpoint

`tauri-apps/tauri-action` has no `configPath` input. Beta config overrides are passed via the `args` input using Tauri's `--config` flag:

```yaml
args: '--target aarch64-apple-darwin --config src-tauri/tauri.beta.conf.json'
```

`tauri.beta.conf.json` must override at minimum:

- `identifier` → `com.gm-tool.beta`
- `productName` → `GM-Tool Beta`
- `plugins.updater.endpoints` → beta-channel endpoint URL

## GitHub Actions workflow structure

Two workflow files, not six:

```text
.github/workflows/
  release.yml           # triggers on v[0-9]+.[0-9]+.[0-9]+ tags (no pre-release suffix)
  beta-release.yml      # triggers on v*-beta* tags
```

Each file uses a `matrix` strategy with three entries — one for Windows and two for macOS (Apple Silicon and Intel). macOS requires two separate matrix entries because building a universal binary via `--target universal-apple-darwin` is not the recommended path through tauri-action:

```yaml
strategy:
  fail-fast: false
  matrix:
    include:
      - os: windows-latest
        args: ''
        target: ''
      - os: macos-latest
        args: '--target aarch64-apple-darwin'
        target: 'aarch64-apple-darwin'
      - os: macos-latest
        args: '--target x86_64-apple-darwin'
        target: 'x86_64-apple-darwin'
```

### Signing environment variables

`TAURI_SIGNING_PRIVATE_KEY` and `TAURI_SIGNING_PRIVATE_KEY_PASSWORD` must be set at the **job level**, not the step level. The Tauri CLI reads them when bundling — if they are only scoped to the tauri-action step they may not reach the subprocess that does the actual signing:

```yaml
jobs:
  release:
    env:
      TAURI_SIGNING_PRIVATE_KEY: ${{ secrets.TAURI_SIGNING_PRIVATE_KEY }}
      TAURI_SIGNING_PRIVATE_KEY_PASSWORD: ${{ secrets.TAURI_SIGNING_PRIVATE_KEY_PASSWORD }}
```

`GITHUB_TOKEN` stays on the tauri-action step only — it is used for release asset uploads, not for signing.

### Trigger patterns

`release.yml`:

```yaml
on:
  push:
    tags:
      - 'v[0-9]+.[0-9]+.[0-9]+'
```

`beta-release.yml`:

```yaml
on:
  push:
    tags:
      - 'v*-beta*'
```

### Key differences between the two workflow files

| | `release.yml` | `beta-release.yml` |
|---|---|---|
| Trigger tag pattern | `v1.2.3` (no suffix) | `v1.2.3-beta.N` |
| `prerelease` flag in tauri-action | `false` | `true` |
| `args` in tauri-action | platform target only | platform target + `--config src-tauri/tauri.beta.conf.json` |
| "Verify tag is on main" step | present | absent |

Shared setup steps (checkout, Node, Rust toolchain, `npm install`) are identical in both files. When updating action versions or Node version, both files must be updated.
